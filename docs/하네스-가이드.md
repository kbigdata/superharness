# 슈퍼하네스(superharness) 전체 구조 가이드

> 이 문서는 **처음 보는 사람이 한 번에 전체 그림을 잡도록** 쓴 안내서입니다.
> 코드를 따라가며 "무엇이 / 왜 / 어떻게 / 어디서"를 연결합니다.
> 기준 버전: **v0.2.0** · Python 3.11+ · 기본 오프라인(MockProvider)

---

## 0. 30초 요약

**슈퍼하네스**는 "여러 AI 에이전트를 줄 세워 일을 끝까지 시키는" **멀티에이전트 오케스트레이션 하네스**입니다.
특정 프레임워크(Claude Code 등)에 묶이지 않은 **독립 실행형 Python 코어**이고, 키 없이 오프라인으로도 전부 돌아갑니다.

핵심은 4개의 패턴입니다.

| # | 패턴 | 한 줄 설명 | 코드 위치 |
|---|------|-----------|-----------|
| 1 | **키워드 → 스킬 활성화** | 프롬프트의 트리거 단어를 감지해 워크플로(스킬)를 켠다 | `skills/` |
| 2 | **멀티에이전트 오케스트레이션** | plan→exec→verify→fix로 에이전트를 분업·병렬 실행 | `orchestration/`, `agents/` |
| 3 | **영속성(Ralph) 루프** | 검증될 때까지 멈추지 않고 verify→fix 반복 | `orchestration/ralph.py`, `hooks/` |
| 4 | **상태 + 티어 라우팅** | 작은 메타(control)와 큰 산출물(data)을 분리, 난이도→모델 라우팅 | `state/`, `config.py`, `providers/` |

## 1. 큰 그림 — 레이어 지도

```
                          ┌─────────────────────────────────────────┐
   사용자 프롬프트  ──────▶ │  ① 스킬 레이어 (skills/)                  │
   "ralph: CSV 파서 …"     │  KeywordDetector → SkillInjector          │
                          │  = 어떤 모드로, 어떤 컨텍스트를 주입할지   │
                          └───────────────┬─────────────────────────┘
                                          │ injected_context + mode
                                          ▼
                          ┌─────────────────────────────────────────┐
                          │  ② 오케스트레이션 (orchestration/)        │
                          │  TeamPipeline: plan→exec→verify→fix       │
                          │   ├ Orchestrator (병렬 워커, TaskList)    │
                          │   └ RalphLoop   (③ 영속 루프)             │
                          └───────┬───────────────────────┬─────────┘
                                  │ agent.run()            │ STOP 이벤트
                                  ▼                        ▼
              ┌──────────────────────────┐   ┌──────────────────────────┐
              │  에이전트 (agents/)       │   │  훅 버스 (hooks/)         │
              │  도메인×티어 매트릭스     │   │  PersistentMode = STOP 가드│
              └────────────┬─────────────┘   └──────────────────────────┘
                           │ CompletionRequest(model=…)
                           ▼
              ┌──────────────────────────┐   ┌──────────────────────────┐
              │  ④ 프로바이더 (providers/)│   │  ④ 상태 (state/)          │
              │  mock | anthropic         │   │  control plane: StateStore │
              │  Tier→모델 해석           │   │  data plane:   ArtifactStore│
              └──────────────────────────┘   └──────────────────────────┘
```

**핵심 원리**: 위에서 아래로 의존이 흐릅니다. 윗 레이어는 추상(Protocol/역할명)만 알고,
구체 구현(어떤 모델, 어떤 백엔드)은 맨 아래에서 교체됩니다. → 그래서 오프라인 mock ↔ 실제 anthropic을 한 줄로 갈아끼웁니다.

## 2. 전체 워크플로우 — `superharness demo` 한 번에 따라가기

가장 대표적인 E2E 흐름입니다. `cli.py`의 `demo` → `_run_pipeline` → `TeamPipeline.run`을 그대로 풀어쓴 것입니다.

```
입력: "ultrawork: build a CSV parser, don't stop until tests pass"

[1] 스킬 활성화      SkillRegistry.load().activate(prompt)
    │                트리거 "ultrawork", "don't stop" 감지
    │                → 활성 스킬: [ultrawork, ralph], 모드: ralph (가장 강함)
    │                → injected_context = <skill>…</skill> 블록들
    ▼
[2] 파이프라인 시작   TeamPipeline.run(goal)
    │                HookBus + PersistentMode 부착 (STOP 차단 준비)
    │                emit(SESSION_START)
    ▼
[3] PLAN            planner(HIGH=opus) 에이전트가 목표를 태스크 목록으로 분해
    │                결과 텍스트를 ArtifactStore에 'plan'으로 기록 → descriptor
    │                "- " 로 시작하는 줄들을 executor 태스크로 변환 (_derive_tasks)
    ▼
[4] EXEC (병렬)     Orchestrator.run(TaskList)
    │                워커 N개가 TaskList에서 claim → executor(MEDIUM) 실행 → complete
    │                동시성은 CapacityLimiter(max_concurrency=4)로 제한
    │                각 산출물은 ArtifactStore에 'result'로 해시 저장
    ▼
[5] VERIFY          qa-tester(MEDIUM)가 모든 산출물을 모아 목표 만족 여부 판정
    │                출력에 "FAIL" 없으면 complete=True
    ▼
[6] FIX 루프(Ralph)  RalphLoop: complete면 종료, 아니면
    │   ┌────────────┐  · 모든 exec 태스크 reopen → Orchestrator 재실행 → 다시 VERIFY
    │   │ 미완료 시  │  · 최대 max_iterations(10)회까지 반복
    │   │ STOP 차단  │  · PersistentMode가 "미검증" 동안 STOP 이벤트를 block
    │   └────────────┘
    ▼
[7] 종료            검증 완료 시 mark_complete → STOP 통과 → emit(SESSION_END)
                    PipelineResult(plan, results[], verified, iterations) 반환
```

출력 예시(개념):
```
활성 스킬: ['ultrawork', 'ralph'] / 모드: ralph
목표: ultrawork: build a CSV parser, don't stop until tests pass
플랜: artifacts/<sha>.md
결과 아티팩트: 3개
검증 완료: True (반복 2)
```

## 3. 패턴별 상세

### ① 키워드 → 스킬 활성화 (`skills/`)

**무엇**: 자연어 프롬프트에서 트리거 단어를 찾아, 대응하는 "스킬"(워크플로 지침)을 켭니다.

**스킬이란?** YAML frontmatter + 마크다운 본문으로 된 `.md` 파일입니다.

```markdown
---
name: ralph
description: 검증 완료까지 멈추지 않는 지속/검증 루프
triggers: ["ralph", "don't stop", "must complete", "until done"]
mode: ralph                # plain < ultrawork < autopilot < team < ralph
pipeline: ["planner", "executor", "qa-tester"]   # (선택)
---
목표가 검증으로 완료될 때까지 verify→fix 루프를 반복한다. …(본문 = 주입될 지침)
```

**흐름** (`SkillRegistry.activate`):
```
prompt
  → KeywordDetector.detect()     # \b단어\b 정규식, 긴 트리거 우선, 스킬당 1회
  → [MatchedTrigger…]
  → SkillInjector.activate()
      · 매칭 스킬 본문을 <skill name="…">…</skill> 블록으로 감쌈 → injected_context
      · 여러 스킬이 켜지면 _MODE_RANK로 가장 강한 mode 채택
  → Activation(skills, injected_context, mode, pipeline)
```

**스킬 로딩 우선순위** (`SkillRegistry.load`, 뒤가 앞을 override):
```
builtin(패키지 내장)  <  ~/.superharness/skills (유저)  <  ./.superharness/skills (프로젝트)
```
내장 스킬 5종: `autopilot`, `ralph`, `team`, `ultrawork`, `karpathy`.

**자동 스킬 생성/관리 (v0.2.0의 핵심 확장)** — "한 번 잘한 일을 재사용 스킬로 굳히기":
```
검증 통과 세션
  → SkillLearner.learn(goal, trace, verified=True)     # Goal-Driven 게이트: 검증된 것만
       · learner(HIGH) 에이전트가 재사용 패턴 1개 추출
       · karpathy 4원칙을 injected_context로 자동 주입
       · (옵션) semantic_judge: LLM이 의미 중복 판정
  → SkillWriter.propose(md)                            # 4단 게이트
       1) 파싱 검증     (frontmatter 깨지면 거부)
       2) 안전성 스캔   (api_key/secret/rm -rf/curl http… deny-list)
       3) dedup        (이름 충돌 / 트리거 충돌 / 어휘 유사도≥0.6)
       4) 격리 기록     → .superharness/skills-proposed/  (자동 로드 안 됨!)
  → 사람이 검토 후  `skills promote <name>`             # 활성 디렉토리로 승격 + 버전 기록
```
> **안전 설계**: 자동 생성 스킬은 **신뢰 불가**로 취급 → 항상 격리(quarantine). 사람이 promote해야만 활성화됩니다.

**버전 관리 / 자기개선** (`SkillVersionStore`, `SkillLearner.refine`):
- `skills refine <name>` → critic(HIGH)이 기존 스킬을 다듬은 후보를 다시 격리(제안).
- promote/rollback 할 때마다 내용을 content-hash 아티팩트 + JSON 인덱스로 기록.
- `skills history <name>` / `skills rollback <name> <version>` → 드리프트 시 되돌리기.

#### 왜 "Hermes식"인가 — 자기개선 루프의 계보

`learner.py`는 자신의 방식을 **"Hermes식 자동 스킬 생성의 안전한 형태"**라고 부릅니다.
여기서 **Hermes**는 "에이전트가 스스로 한 일에서 재사용 가능한 스킬/도구를 만들어 다음 작업에 쓰는
**자기개선(self-improving) 에이전트** 계열"을 가리키는 이름입니다. 슈퍼하네스는 그 아이디어를 빌리되,
**무방비 자동화의 위험**(잘못된/위험한 스킬이 그대로 활성화되는 것)을 막는 안전판을 끼웠습니다.

| 순수 "Hermes식" 자기개선 | 슈퍼하네스의 안전형 |
|--------------------------|---------------------|
| 세션 → 스킬 추출 → **즉시 활성화** | 세션 → 추출 → **격리(quarantine)** → 사람이 `promote`해야 활성화 |
| 검증 없이도 학습 | **Goal-Driven 게이트**: 검증 통과 세션만 학습(`verified=True`) |
| 안전성 검사 없음 | deny-list·dedup·파싱 **4단 게이트** 통과해야 격리 |
| 드리프트 누적 | **버전·롤백**으로 되돌리기 가능 |

즉 "스스로 도구를 만든다"는 동력은 가져오되, **활성화 권한은 사람에게** 남겨 둔 형태입니다.

#### 카파시(Karpathy) 4원칙 — 추출기에 자동 주입되는 행동 규율

스킬을 추출/정제할 때(`SkillLearner.learn`·`refine`) learner/critic 에이전트의 system 프롬프트
앞에 **`karpathy` 스킬 본문이 자동 주입**됩니다(`guidance_skill="karpathy"`, `_guidance_context()`).
목적은 추출되는 스킬이 "장황하고 과한" 게 아니라 **간결·정확·검증가능**하도록 규율을 거는 것입니다.
내장 `skills/builtin/karpathy.md`의 4원칙(출처: A. Karpathy, 2026-01):

1. **Think Before Coding (신중하게)** — 가정을 명시한다. 불확실하면 추측하지 말고 **묻는다**.
   여러 해석이 가능하면 모두 제시하고, 더 단순한 방법이 있으면 말한다. 혼란스러우면 멈추고
   무엇이 불분명한지 이름 붙인다.
2. **Simplicity First (간결하게)** — 문제를 푸는 **최소 코드만**. 요청하지 않은 기능·추상화·
   유연성·불가능한 시나리오용 방어 코드를 넣지 않는다. 200줄이 50줄로 가능하면 다시 쓴다.
3. **Surgical Changes (외과적으로)** — **꼭 필요한 곳만** 건드린다. 인접 코드/포매팅을 임의로
   "개선"하지 않고, 망가지지 않은 것을 리팩터하지 않는다. 기존 스타일을 따르고, 죽은 코드는
   삭제하지 말고 언급만 한다.
4. **Goal-Driven Execution (목표 주도)** — 작업을 **검증 가능한 목표**로 바꾼다.
   "검증 추가"→"테스트 작성 후 통과", "버그 수정"→"테스트로 재현 후 수정". 목표가 검증될 때까지 루프한다.

> 4번 **Goal-Driven**은 단순 지침이 아니라 이 하네스의 골격과 직접 맞물립니다 —
> ③ Ralph 루프(검증까지 반복)와 학습의 "검증된 세션만"(Goal-Driven 게이트)이 모두 이 원칙의 구현체입니다.
> `karpathy` 스킬은 모드가 `plain`이라 워크플로를 바꾸지 않고, **가이드(지침)로만** 쓰입니다.

핵심 파일:
- `skills/skill.py` — Skill 모델 + frontmatter 파서
- `skills/keyword_detector.py` — 트리거 감지
- `skills/injector.py` — 컨텍스트 주입 + 모드 결정
- `skills/registry.py` — 로딩/우선순위
- `skills/writer.py` — 게이트/격리/승격/롤백
- `skills/versions.py` — 버전 이력
- `skills/similarity.py` — 어휘 유사도(Jaccard) dedup
- `orchestration/learner.py` — 추출/정제 오케스트레이션

### ② 멀티에이전트 오케스트레이션 (`orchestration/`, `agents/`)

**에이전트 = 도메인 × 티어**. 클래스를 새로 쓰는 게 아니라 **데이터(매트릭스)를 편집**해 추가합니다.

`agents/registry.py`의 `_DEFAULT_MATRIX` (도메인, 티어, 시스템 프롬프트):

| 도메인 | 가능 티어 | 역할 |
|--------|-----------|------|
| `architect` | LOW/MEDIUM/HIGH | 구조·설계 분석 |
| `executor` | LOW/MEDIUM/HIGH | 실제 구현 |
| `explore` | LOW/HIGH | 정보 탐색 |
| `designer` | MEDIUM | UI 설계 |
| `planner` | HIGH | 목표→태스크 분해 |
| `critic` | HIGH | 결함 비판/정제 |
| `qa-tester` | MEDIUM | 산출물 검증 |
| `security-reviewer` | HIGH | 취약점 점검 |
| `learner` | HIGH | 세션→스킬 추출 |

- 이름 규칙: MEDIUM은 도메인명 그대로(`executor`), 그 외엔 `executor-high` 식.
- `get(domain, tier)`는 없으면 **인접 티어로 폴백**(MEDIUM→HIGH→LOW).
- 한 에이전트의 실행(`agents/agent.py` `Agent.run`):
  ```
  티어 → 모델 해석(tiers.resolve)
  system = 시스템프롬프트 + injected_context
  HIGH 티어면 thinking=True, effort="high"
  provider.complete(CompletionRequest) → 텍스트
  artifacts.write(...) → ArtifactDescriptor
  SUBAGENT_START / SUBAGENT_STOP 훅 emit
  ```

**TeamPipeline** (`orchestration/pipeline.py`) = 전체 안무가:
```
run(goal):
  emit(SESSION_START)
  plan  : planner(HIGH).run → plan 아티팩트
  derive: 플랜의 "- " 줄 → executor 태스크들 (없으면 goal 1개)
  exec  : Orchestrator.run(TaskList)            # 병렬
  verify: qa-tester(MEDIUM) → "FAIL" 없으면 complete
  fix   : RalphLoop(verify, fix)                # ③로 위임
  return PipelineResult
```

**Orchestrator** (`orchestration/orchestrator.py`) = 병렬 디스패처:
```
워커 코루틴 N개 (N = min(max_concurrency, 대기 태스크 수))
각 워커: while True { task = TaskList.claim(); if None: return; run_one(task) }
동시성: anyio.CapacityLimiter(max_concurrency)
개별 태스크 실패는 격리 → tasklist.fail (전체 중단 안 함)
```

**TaskList** (`orchestration/task.py`) = 공유 컨트롤플레인 큐:
- 상태: `PENDING → CLAIMED → DONE | FAILED`, 그리고 `reopen()`(fix용).
- `claim/complete/fail/reopen`은 **단일 `asyncio.Lock`으로 원자적** → 워커 경합 안전.

### ③ 영속성(Ralph) 루프 (`orchestration/ralph.py`, `hooks/`)

**무엇**: "절반만 하고 멈추지 마라"를 강제하는 장치.

**RalphLoop** — 제네릭 `verify`/`fix` 콜러블을 받아 반복:
```
for i in 1..max_iterations:
    report = verify()
    if report.complete:
        persistent.mark_complete(); emit(STOP, complete=True); return 성공
    persistent.mark_incomplete(); emit(STOP, complete=False)   # ← STOP 차단됨
    fix(report)
return 미완료(max 도달)
```
> 제네릭이라 **Team의 fix-loop**와 **독립 `ralph` 모드** 양쪽을 같은 코드로 굴립니다.

**HookBus + PersistentMode** (`hooks/bus.py`) — 비종료성의 실제 강제 장치:
```
HookBus: async pub/sub. on(event, handler) / emit(event, payload) / fired(event)
LifecycleEvent: SESSION_START, SUBAGENT_START/STOP, STOP, SESSION_END

PersistentMode: STOP 이벤트에 핸들러 등록
    목표 미검증(_complete=False) → HookOutcome(block=True)   # STOP을 막음
    검증 완료(_complete=True)    → HookOutcome(block=False)  # STOP 통과
```
즉 "검증 안 끝났으면 멈출 수 없다"가 **이벤트 레벨에서 보장**됩니다(인프로세스 가드).

---

### ④ 상태 + 티어 라우팅 (`state/`, `config.py`, `providers/`)

**control plane vs data plane 분리** — 작은 메타와 큰 산출물을 따로 둡니다.

```
.superharness/                       ← StateLayout (state/paths.py)
├── state/
│   └── sessions/<id>/               control plane: 세션 메타(작은 JSON)
├── project-memory.json              control plane: 누적 메모리
├── artifacts/<sha256>.<ext>         data plane:   큰 내구 산출물
├── plans/ specs/ notepads/ handoffs/
└── skill-versions.json              스킬 버전 인덱스
```

- **StateStore** (`state/store.py`): 작은 JSON(세션, project-memory) 읽기/병합.
- **ArtifactStore** (`state/artifacts.py`): 내용을 SHA-256으로 해시 → `artifacts/<hash>.<ext>`에 저장하고
  **ArtifactDescriptor**(kind/path/content_hash/producer/created_at/size)를 반환.
  control plane은 무거운 내용 대신 이 **가벼운 디스크립터만** 들고 다님. `read()` 시 해시 검증으로 무결성 보장.
- **경로 안전** (`state/paths.py`): `ReadPath`/`WritePath`는 검증 생성자(`as_read_path`/`as_write_path`)로만
  만들어지는 `NewType`. 상태 루트를 벗어나는 경로(`../` traversal)는 `PathViolation`으로 차단.
  리더/라이터 시그니처를 갈라 두면 **mypy가 교차 대입까지 거부**.

**티어 → 모델 라우팅** (`config.py`):
```
Tier.LOW    → claude-haiku-4-5     (빠름/저렴)
Tier.MEDIUM → claude-sonnet-4-6    (기본 작업)
Tier.HIGH   → claude-opus-4-8      (어려운 계획/검증/추출)
```
환경변수 `SUPERHARNESS_TIER_{LOW,MEDIUM,HIGH}`로 override 가능.

**프로바이더 경계** (`providers/`) — 유일한 교체 지점:
```
Provider Protocol:  async complete(CompletionRequest) -> CompletionResult
get_provider(name): "mock"(기본·오프라인) | "anthropic"(lazy import)
```
- **MockProvider**: 호출을 기록하고 해석된 모델명을 echo → 네트워크 없이 라우팅/흐름 테스트.
- **AnthropicProvider**: 실제 Claude. **SDK 셰이핑이 load-bearing**:
  - Opus/Sonnet에서 `thinking={"type":"adaptive"}`, effort는 `output_config={"effort":…}`.
  - `temperature`/`top_p`/`budget_tokens`는 **절대 전송 금지**(해당 모델에서 400).
  - effort는 opus/sonnet-4-6에서만, Haiku에선 미전송.
  - `max_tokens`가 크면(>16k) 스트리밍으로 보내 타임아웃 회피.
  - `anthropic` import는 lazy → mock 전용 설치는 의존성 불필요.

## 4. CLI 표면 — 각 단계를 독립 실행

`superharness <command>` (uv 사용 시 `uv run superharness …`). 각 명령은 한 레이어를 그대로 노출합니다.

| 명령 | 하는 일 | 닿는 레이어 |
|------|---------|-------------|
| `ask "<프롬프트>" --tier --provider` | 단일 완성(in→out) | ④ provider |
| `state init` | 상태 트리 생성 + 샘플 아티팩트 | ④ state |
| `skills list` | 로드된 스킬/트리거 나열 | ① |
| `skills detect "<프롬프트>"` | 트리거 감지 + 활성화(모드/주입) 출력 | ① |
| `skills proposed` | 격리된(제안) 스킬 목록 | ① writer |
| `skills promote <name>` | 격리 → 활성 승격 + 버전 기록 | ① writer/versions |
| `skills refine <name> --note` | critic으로 개선안 생성(격리) | ① learner |
| `skills history <name>` | 버전 이력 | ① versions |
| `skills rollback <name> <ver>` | 특정 버전으로 롤백 | ① versions |
| `agents run <name> "<태스크>"` | 단일 에이전트 실행 + 아티팩트 | ② agents |
| `team "<목표>" [--learn]` | 풀 파이프라인(plan→exec→verify→fix) | ②③④ |
| `demo` | 스킬감지→파이프라인→Ralph E2E | 전체 |

> `--provider`를 안 주면 설정값(기본 `mock`)을 따르므로 **키 없이 전부 실행**됩니다.
> 실제 호출은 `SUPERHARNESS_PROVIDER=anthropic` + `ANTHROPIC_API_KEY` 설정 시.

## 8. 핵심 설계 원칙(왜 이렇게 만들었나) 요약

1. **프레임워크 중립**: 모든 LLM 접근은 `Provider` Protocol 하나로 추상화 → 백엔드 교체가 한 줄.
2. **오프라인 우선**: 기본 mock → 키/네트워크 없이 전 기능·전 테스트 실행.
3. **데이터 주도 확장**: 에이전트는 클래스가 아니라 매트릭스 데이터로 추가.
4. **control/data plane 분리**: 작은 메타는 가볍게 돌리고, 큰 산출물은 해시 주소로 내구 저장.
5. **안전 기본값**: 경로 traversal 차단(NewType+검증), 자동 생성 스킬은 격리 후 사람이 승격.
6. **비종료성의 강제**: "검증 전 STOP 금지"를 이벤트 훅 레벨에서 보장(Ralph + PersistentMode).
7. **로케일**: 코드 주석·스킬 본문은 한국어로 통일.

## 8.5. 메모리 & 코드베이스 이해 레이어 (신규)

기존 4대 패턴 위에 "기억"과 "코드 읽기"를 더한 확장. **신규 의존성 0·오프라인 유지.**

### 메모리 (state/)
- **자동 적립** (`MemoryRecorder`): `PersistentMode`와 같은 패턴으로 `HookBus`를 구독해
  `SUBAGENT_STOP`·`SESSION_END`·`PRE_COMPACT` 이벤트를 `project-memory.json`의 `events[]`에
  타임스탬프와 함께 자동 누적. `_run_pipeline`에 한 줄로 부착돼 **에이전트가 일하는 동안 기억이 쌓인다**.
- **구조화 메모리 + 회상** (`MemoryStore`/`MemoryInjector`): `memory.json`에 `MemoryEntry`
  (namespace·tags·created_at). 오프라인 검색(`query`: ns/tag/since/부분문자열) + 토큰 Jaccard
  랭킹 회상(`recall`). 한국어 조사/어미는 **CJK 문자 bigram**으로 흡수. 파이프라인은 목표와
  관련된 메모리를 `<memory>` 블록으로 스킬 컨텍스트와 함께 주입.
- CLI: `memory add/query`.

### 코드베이스 이해 (tools/)
- **읽기 전용 코드베이스** (`Codebase`): `glob`/`read`/`grep` — 표준 라이브러리만, `as_read_path`로
  traversal 차단, 바이너리·노이즈 디렉토리(.git 등) skip. `context_for(query)`는 단서 라인을
  `<codebase>` 블록으로 만들어 주입(Provider 변경 0).
- **코드맵** (`CodeMap`, deepinit식): 파이썬 `ast`로 파일별 top-level 클래스/함수 심볼을 추출해
  마크다운 코드맵 생성. (정밀 LSP/AST는 후순위·선택 extra로 분리)
- 신규 에이전트 도메인 `code-explorer`(MEDIUM) — 매트릭스 한 줄 추가.
- CLI: `codebase glob/read/grep/map`.

### 멀티프로젝트 격리 (config.py)
- `SUPERHARNESS_STATE_ROOT` 설정 시 effective 상태경로 = `{state_root}/{project_id}`.
  `project_id` 미설정이면 cwd 해시로 자동 파생. 미설정 시 기존 `./.superharness` 그대로(하위호환).

### 위키 / 세션 검색 (state/)
- `WikiStore`: `notepads/wiki.md`에 세션 누적 KB(`append`/`render`/`sections`).
- `StateStore.search_sessions(query, since=)`: 세션 meta.json 부분문자열/시간 검색.
- CLI: `wiki add/show`, `session search`.

> 설계 근거·OMC 비교는 로컬 전용 문서(`docs/메모리-코드베이스-비교.md`,
> `docs/개선계획-메모리-코드베이스.md`) 참고.
